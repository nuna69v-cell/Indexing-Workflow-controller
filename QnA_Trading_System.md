# Trading System Q&A

## 1. How to maintain all repositories in a good state and controllable amount of branches?
To maintain multiple repositories efficiently:
- **Master Map Architecture**: Create a `.master-map.json` that acts as an index of all your repositories. You can use a Python script (like `sync-all.py`) to crawl your GitHub accounts (e.g., mouyleng172, LengKundee) and list all repository URLs, branches, and statuses.
- **Automated Syncing**: Use the `sync-all.py` script to automate Git operations (`git fetch`, `git pull`, `git status`). Instead of checking each repo manually, the script can tell you which repos have uncommitted changes or are out of sync.
- **Branch Management Policy**: Adopt a simple branching strategy like GitHub Flow. Keep the `main` or `master` branch always deployable. Create feature branches for new work (`feature/new-indicator`) and delete them after merging. Limit active branches per repo to 2-3 to reduce clutter.
- **Git Config Management**: If working with multiple accounts on one machine, use conditional includes in your `.gitconfig` to automatically apply the correct SSH keys and credentials based on the folder path.

## 2. How can we make all repositories continuously running for supporting the trading system for free on Exness-MT5Real24 and FxPro-MT5 with wallet setup broker connection to get commission from the provided broker?
Running a 24/7 trading system for free requires leveraging free tier cloud services:
- **Cloud Providers**: Use services like Google Cloud Platform (e2-micro instance), AWS EC2 (t2.micro), or Oracle Cloud (Always Free ARM instances) to host your Python backend and scheduling scripts.
- **Broker Connections**: Use the MetaTrader 5 (MT5) Python integration (`MetaTrader5` library). You need to install the MT5 terminal on a Windows VPS (or using Wine on Linux, though Windows is much more stable for MT5). Since free cloud tiers usually provide Linux, you might need to use a Dockerized MT5 setup (like the `exness-docker` setup in your project) or connect a local Windows machine running MT5 to your cloud backend via API.
- **Commissions/IB (Introducing Broker)**: To get commissions, you must register as an IB/Partner with Exness and FxPro. You will receive an affiliate link. You must open your trading accounts under this link, or refer others. The broker will automatically deposit commissions into your partner wallet based on the trading volume generated by your system.
- **Architecture**: Your cloud backend (FastAPI/Node.js) generates signals and sends them to the MT5 terminal via a webhook/API or by writing to a shared file/database that an Expert Advisor (EA) running in MT5 constantly reads.

## 3. How can we scalping trade?
Scalping involves making numerous trades to capture small price movements:
- **Low Latency**: Speed is critical. Your system must be hosted geographically close to the broker's trading servers (e.g., if Exness servers are in London, host your VPS in London).
- **Tight Spreads**: Use raw spread or zero spread accounts (like Exness Raw Spread account) to minimize transaction costs.
- **Strategy Logic**: Focus on shorter timeframes (M1, M5). Your Python logic should analyze order book imbalances, tick data, or fast-moving indicators (like stochastic, EMA crosses on short periods).
- **Execution**: The EA must execute trades instantly. Avoid complex calculations inside the EA at the moment of execution. The Python backend should pre-calculate signals and the EA should only be responsible for executing the `OrderSend` command as fast as possible.

## 4. How to improve speed execution strategy and indicator of the trading system?
- **Tick Data Processing**: Instead of waiting for candle close, process tick data. This requires a fast data pipeline.
- **Compiled Languages for Heavy Math**: If your indicators use heavy math, consider writing the core logic in C++ or Cython and wrapping it for Python, or use optimized libraries like NumPy and Pandas.
- **Asynchronous Operations**: Ensure your Python backend uses `asyncio` for all network requests (fetching data, sending signals) so it doesn't block while waiting for responses.
- **Memory Caching**: Use Redis to store recent price data and indicator values instead of querying a database constantly.
- **Optimize EA**: In MQL5, ensure your EA is optimized. Minimize `Print()` statements, avoid unnecessary loops in `OnTick()`, and use asynchronous order sending (`OrderSendAsync`) if possible.

## 5. How can we set up a schedule for Jules and automate?
- **Jules CLI**: Ensure Jules is installed globally via npm (`npm install -g @google/jules`).
- **Scheduling Script**: The project contains `amp_scheduler.py` which uses the `schedule` library. This script acts as the main engine for triggering trading jobs at specific times.
- **Background Process**: Run the scheduler as a background process or a Windows Service (using `genx_24_7_service.py` or tools like PM2 or `nohup` on Linux).
- **Automation Pipeline**: The scheduler triggers `amp_job_runner.py`, which executes the trading pipeline: collecting market data, gathering sentiment, generating AI predictions, creating signals, and sending them to the execution module.

## 6. How to set up a repository in GitHub to fit the strategy of the code?
- **Monorepo vs. Polyrepo**: A monorepo (like your current structure) is easier for managing dependencies across the backend, frontend, and EAs.
- **Directory Structure**: Keep distinct components separated:
  - `/api` or `/backend`: Python FastAPI code, AI models, and scheduling logic.
  - `/client` or `/frontend`: React dashboard UI.
  - `/expert-advisors`: MQL4/MQL5 code for the trading terminals.
  - `/docs`: Documentation and setup guides.
  - `/scripts`: Deployment and maintenance scripts.
- **CI/CD**: Set up GitHub Actions (`.github/workflows`) to automatically run tests (`pytest`) when you push to the `main` branch.
- **Secrets Management**: Store API keys, Database URLs, and VPS credentials in GitHub Secrets, NEVER in the code.
- **Branching**: Protect the `main` branch. Require pull requests to merge code, ensuring all new strategies are reviewed before going live.
